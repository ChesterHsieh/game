## Implementation Notes

*Derived from ADR-001 and GDD Detailed Design §Core Rules 8–9, §UI Requirements:*

**Full `_unhandled_input` implementation (GDD Core Rule 8)**:
```gdscript
func _unhandled_input(event: InputEvent) -> void:
    # State guard — implemented in Story 002
    if _state == State.REVEALING or _state == State.BLACKOUT:
        return
    # Only process input in Holding state
    if _state != State.HOLDING:
        return
    # Reject mouse motion (always)
    if event is InputEventMouseMotion:
        return
    # Reject mouse button release
    if event is InputEventMouseButton and not event.pressed:
        return
    # Reject key release, echo keys, and KEY_ESCAPE
    if event is InputEventKey:
        if not event.pressed or event.echo:
            return
        if event.keycode == KEY_ESCAPE:
            return
    _on_dismiss()
```

**Dismiss handler (GDD Core Rule 9)**:
```gdscript
func _on_dismiss() -> void:
    _state = State.QUITTING
    get_tree().quit()
```
`get_tree().quit()` is called exactly once. No scene swap. No return to Main Menu. The application window closes. `_state = State.QUITTING` is set first as a guard against any theoretical re-entrant call — though Godot's single-threaded dispatch makes re-entrancy impossible in practice.

**Why KEY_ESCAPE is explicitly rejected (GDD EC-8)**: Esc is the OS-level quit shortcut on some platforms and is mapped to "quit" on the Main Menu. Allowing Esc to dismiss FES creates a "double-tap-to-quit" failure mode where a user who reflexively hits Esc on a previous screen could dismiss FES before seeing the illustration. The explicit `keycode == KEY_ESCAPE` rejection guard is a design decision, not a tuning knob.

**Why mouse motion is always rejected (GDD EC-5)**: `InputEventMouseMotion` fires continuously as the cursor moves. Even with cursor hidden (`Input.MOUSE_MODE_HIDDEN`, set in Story 004), mouse motion events can still be generated by hardware. Accepting them would make the image dismiss itself the moment any mouse movement occurs — including involuntary hand movement during reading.

**Why button/key release is rejected (GDD EC-7, EC-6)**: A user who held a key or mouse button during the blackout window would generate a `pressed=false` release event immediately after the blackout expires. Accepting release events would cause an instant dismiss the moment the blackout ends. Only fresh `pressed=true` events (post-blackout) are valid dismissals.

**Why `echo=true` is rejected (GDD EC-6)**: A key held down during the blackout continues generating echo events after the blackout expires. These would cause repeated quit calls (though `State.QUITTING` guard prevents duplicates). Echo rejection ensures only intentional, deliberate key presses count.

**No signal on dismiss**: FES is a terminal leaf node (GDD §Dependencies — Downstream: None). `get_tree().quit()` is the complete dismiss action. No EventBus signal is emitted; no other system needs to react.

## QA Test Cases

*Logic — automated (`tests/unit/final-epilogue-screen/fes_input_filter_test.gd`):*

- **AC-INPUT-3**: Valid dismiss in Holding → `get_tree().quit()` called exactly once
  - Given: FES state advanced to Holding (blackout Timer has timed out); quit spy connected
  - When: simulate `InputEventKey` with `pressed=true, echo=false, keycode=KEY_SPACE`
  - Then: `get_tree().quit()` is called exactly once; FES state is Quitting
  - Edge cases: simulate a second valid input immediately after first dismiss — `get_tree().quit()` must NOT be called a second time (Quitting state guard); simulate `InputEventMouseButton` with `pressed=true` (left button) in Holding — must also dismiss exactly once

- **AC-INPUT-4**: `InputEventMouseMotion` never dismisses
  - Given: FES state is Holding
  - When: simulate `InputEventMouseMotion` continuously for 5 seconds (100 events)
  - Then: `get_tree().quit()` is never called; FES state remains Holding throughout
  - Edge cases: simulate `InputEventMouseMotion` in Revealing state — must also be ignored (state guard from Story 002 covers this, but test should confirm)

- **AC-INPUT-5**: `KEY_ESCAPE` never dismisses
  - Given: FES state is Holding
  - When: simulate `InputEventKey` with `pressed=true, echo=false, keycode=KEY_ESCAPE`
  - Then: `get_tree().quit()` is NOT called; FES state remains Holding
  - Edge cases: simulate `KEY_ESCAPE` with `pressed=false` (release) — must also be rejected (would be caught by the `not event.pressed` guard before reaching the keycode check)

- **AC-INPUT-6**: Mouse button release never dismisses
  - Given: FES state is Holding
  - When: simulate `InputEventMouseButton` with `pressed=false` (button release, any button)
  - Then: `get_tree().quit()` is NOT called; FES state remains Holding
  - Edge cases: simulate left-button release, right-button release, middle-button release — all must be rejected

- **AC-INPUT-7**: Key echo (`echo=true`) never dismisses
  - Given: FES state is Holding
  - When: simulate `InputEventKey` with `pressed=true, echo=true, keycode=KEY_SPACE`
  - Then: `get_tree().quit()` is NOT called; FES state remains Holding
  - Edge cases: simulate echo event with `echo=true, pressed=true` for multiple keycodes (KEY_ENTER, KEY_SPACE, KEY_A) — all must be rejected
